Content Sync — Sanity → Storefront

Define the Sanity content model, deliver editorial content and content assets to the Mosaik Storefront, and specify how editorial content combines with catalog data on shared pages.

User Stories
  • 1
    As Pretec content team, I want to author and publish editorial landing pages in Sanity and see them live on the storefront without requiring a developer deployment.
  • 2
    As Pretec content team, I want to author category banners and descriptions in Sanity that appear on the storefront category pages alongside product listings from the catalog.
  • 3
    As Pretec content team, I want to manage the storefront navigation / mega menu in Sanity so I can update it without a code change.
  • 4
    As a storefront shopper, I want editorial category content and product catalog to appear together on category pages so I get a cohesive experience.
  • 5
    As Pretec content team, I want a clear boundary between what I manage in Sanity vs. Struct so I don't duplicate content or maintain it in two places.
Implementation Tasks
  • Confirm which content types are in Phase 1 scope with Pretec
  • Define Sanity content model / schema per agreed content type
  • Define the boundary between Sanity (editorial) and Struct (product enrichment) — avoid overlap
  • Design how Sanity content and catalog data combine on shared pages (e.g., category page = catalog + Sanity block)
  • Decide delivery mechanism — Sanity API (runtime), webhook-triggered build, or build-time static
  • Implement preview / draft / publish workflow in Sanity
  • Define asset handling and CDN delivery for content images
  • Define localization of content (ties to X-1 Localization — NO Phase 1 only)
  • Confirm who authors and approves content (Pretec roles)
  • Integrate Sanity content delivery into the storefront rendering layer
  • Set up and test initial content in Sanity and validate rendering in storefront
