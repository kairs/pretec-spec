RamBase API Integration — Auth, Credentials, Scoping & Environments

Define and implement the shared foundation that all Geta services use to authenticate to and call the RamBase API — OAuth2 credential model, customer scoping, environment mapping, and operational resilience.

Status:  Partial — consolidating Type: Integration (INT) Shared by: Price, Cart, Quote, Query, Customer write-back (I-4)

A single system/integration credential is used; customer scoping is done via $filter / $db parameters, not per-user tokens.

Blocked Items (require RamBase credentials)
These items are unresolved until Pretec provides RamBase API access:
  • Customer-specific price lookup endpoint for (product, customer, qty)
  • Exact $filter field paths for date/status/PO; Sales Invoice (CIN) + credit notes
  • Create-quote endpoint (CRQ→CQU→COA) and required fields
  • Whether a single system account may read prices for any customer by parameter
User Stories
  • 1
    As Geta engineering, I want a single, clearly documented credential model for all RamBase API calls so every service uses the same auth pattern and I don't re-implement it per service.
  • 2
    As Geta engineering, I want credentials stored and rotatable per environment (test / staging / production) so I can rotate without code changes.
  • 3
    As Geta engineering, I want customer-scoped reads (price, orders, invoices) to work via agreed parameters so I can fetch data for the correct customer without per-user credentials.
  • 4
    As Geta engineering, I want documented retry/timeout behavior for RamBase API calls so all services degrade gracefully when RamBase is slow or unavailable.
  • 5
    As Pretec, I want RamBase API credentials to be properly scoped and stored so the system cannot be exploited to access data beyond its intended scope.
Implementation Tasks
  • Document single system/integration credential model and required OAuth2 scopes
  • Define credential storage solution per environment (secret manager / vault) with rotation procedure
  • Document customer-scoping pattern — $filter on customer field, $db for company selection
  • Map RamBase environments (test, production) to Geta environments (ties to X-2 Environments)
  • Define rate limits, retry strategy, and timeout values for all RamBase API calls (ties to X-4 Resilience)
  • Document document types in use: ART, AGR, CRQ, CQU, COA, CUS, IPA, PLI
  • Resolve blocked items once Pretec provides RamBase API credentials (price endpoint, filter paths, create-quote)
  • Implement credential retrieval and token-refresh in the Pretec Service API
  • Validate auth model in test environment before production rollout
